## Architecture
- Scope: ref/src
- Function: React source for the Pet Manager desktop bridge configurator, split between setup, dashboard, first-level pet-album, and component-center shells, including a responsive component-center app store for negative-screen widgets with unified builtin/draft/create card geometry, install-time button-function customization across screen touch/top red button/front encoder, USB/SSH binding override downlink, coding-agent component generation prompts, generated draft install/delete flows, a pet-album ŌĆ£µ¢░Õ╗║Ķć¬Õ«Üõ╣ēÕĮóĶ▒ĪŌĆØ source chooser for ÕøŠńēćAIńö¤µłÉ / Ķć¬Õ«Üõ╣ēõĖŖõ╝ĀĶ¦åķóæ / Codex Õ»╝Õģź / ńżŠÕī║Õ»╝Õģź, direct MP4-upload custom appearances with all known state slots pre-created for later replacement, per-state appearance WAV prompt-sound configuration that compresses uploads to board-safe 16 kHz mono PCM, lets built-in and custom appearances store user overrides, persists a pending audio-OTA reminder after cue upload/removal with a top-right ŌĆ£õĖŗÕÅæķ¤│µĢłŌĆØ action backed by serialized Tauri appearance asset OTA, supports a preview-first appearance detail workspace with a nearby horizontal state rail that shows every known state, sticky current-state controls, direct per-state MP4 replacement, and a lower contextual info drawer plus custom appearance single-state image+prompt regeneration from a closable detail-page modal that reuses the full custom-avatar wizard upload/config steps, shows an inline generation progress bar after the user starts generation, keeps generation running after the modal is closed, replaces only the selected family video locally, and then reuses the existing appearance asset OTA to replace it on the board, and custom full-avatar generation skips importer-only directional `touch.right` / `touch.left` states while retaining them for imported/legacy records. It falls back to default `done.wav`, `error.wav`, and `waiting_user.wav` prompt cues, includes device guide copy for button customization, a dashboard status strip with manual USB serial rescan plus independent USB/WiFi status chips, shared form-control primitives, and a dashboard button-config area that gives the device callout SVG more room while keeping board/negative-screen button editing compact and sends the full visible button map through USB `button_config` OTA via `button_config_signal`, which only returns after board `button-config-ack` and retries once after reconnecting a stale USB writer. The dashboard voice section keeps session continuation, mock voice-text injection into the active agent session through the managed bridge, and board-originated voice action result status inside the shared shell using the same form rhythm. A new `shell/` subfolder hosts cross-page primitives (PageShell, Card, ContextRail, ToastProvider/useToast/ToastStack, DeviceContextProvider) consumed by `App.jsx`; DeviceContextProvider hydrates active channel state and current desktop target id from the bridge profile before localStorage cache so stale browser state cannot falsely mark an unsynced desktop-pet appearance as current, reconciles stale online-board `targetDeviceId` as well as `targetSource`, owns the appearance USB sync progress so dashboard tab navigation does not reset an in-flight downlink display, while keeping WiFi availability separate from USB direct status for display and routing decisions.
- Sync: if this folder changes, update this file immediately. Plan 3 (2026-05-26) migrated `AppearanceGallery.jsx` onto the shared shell ŌĆö header chrome flows through `<PageShell>`, USB/online/agent state through `useDeviceContext()`, and sync notifications through `useToast()`. On 2026-05-28, `DeviceDashboard.test.js` added coverage that wireless channel switching still reaches MQTT when a stale USB write reports `Connection lost`. On 2026-06-01, widget (negative-screen) USB OTA became ack-gated end-to-end ŌĆö `install_widget_clawpkg` (src-tauri/usb_serial.rs) now waits for board `widget-install-ack` so the ComponentCenter success toast lines up with on-device activation; this is transparent at the React layer, but documented here to clarify why component-center install UX now feels synchronous instead of fire-and-forget. On 2026-06-29, pet-album creation moved behind a four-source chooser and direct MP4 uploads now create replaceable custom appearance state slots; `AppearanceDetail.jsx` shows all known states and lets users upload an MP4 to replace the selected state before using the existing appearance OTA.
- Reuse: `App.jsx` owns cached top-level shell switching; `ComponentCenter.jsx` consumes `BUILTIN_COMPONENT_CENTER` from `fixtures.js` for the builtin component catalog, `lib/component-generation-template.js` for agent prompt payloads, and `lib/clawpkg-contract.js` for the `.clawpkg` bundle contract and `COMPONENT_DASHBOARD_V1` slot schema; `DeviceSetup.jsx` and `DeviceDashboard.jsx` keep operational Tauri flows; appearance screens reuse shared helpers under `lib/`; `DeviceGuideModal.jsx` reads its 3-card content from `lib/device-guide-content.js` (screen mappings plus component-center button-function customization for screen touch, top red button, and front encoder); gallery and component-center action buttons reuse the shared accent/text-on-accent tokens in `styles.css`.
